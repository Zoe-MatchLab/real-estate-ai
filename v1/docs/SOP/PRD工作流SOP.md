## 三、文档标准

### 3.1 PRD文档标准

**定位**：给需求提出人看的解决方案，包含需求背景、实现效果

**篇幅**：建议 500-1500 字

**结构**：

```markdown
# [模块名] PRD

> 版本：1.0  
> 日期：2026-04-13  
> 状态：[草稿/评审中/定稿]

## 1. 背景与目标

### 1.1 需求背景
- 描述当前存在的问题或痛点

### 1.2 目标
- 期望达到的效果

## 2. 用户故事

| ID | 用户故事 | 验收标准 |
|----|---------|---------|
| US-01 | ... | ... |

## 3. 功能概述
- 核心功能说明
- 与其他模块的关联

## 4. 业务规则
- 重要业务逻辑
- 边界条件

## 5. 验收标准

| 功能点 | 验收条件 | 优先级 |
|-------|---------|-------|
| ... | ... | P0/P1/P2 |

## 6. 产品设计

> 详细设计见：[产品设计文档](./design/模块名-产品设计.md)

### 6.1 核心页面
- 简要说明核心页面功能

### 6.2 交互流程
- 关键交互流程简述

## 7. 原型展示

> 交互原型：[原型文件名.html](../prototypes/模块名-原型.html)

### 7.1 页面预览
[可选：关键页面的截图或描述]

### 7.2 交互说明
- 重要交互细节

---

**附件**：
- 产品设计：./design/模块名-产品设计.md
- 交互原型：./prototypes/模块名-原型.html
```

## 四、Review流程

### 4.1 Review轮次

```
PRD Review（PM/需求提出人）
       │
       ▼
   通过？ ───否──▶ 修订 ───▶ 再次Review
       │
      是
       ▼
产品设计 Review（UI/UX）
       │
       ▼
   通过？ ───否──▶ 修订 ───▶ 再次Review
       │
      是
       ▼
原型 Review（FE/需求提出人）
       │
       ▼
   通过？ ───否──▶ 修订 ───▶ 再次Review
       │
      是
       ▼
    定稿
```

### 4.2 Review检查清单

**PRD Review**：
- [ ] 需求背景清晰
- [ ] 用户故事完整
- [ ] 验收标准可测试
- [ ] 无矛盾需求

**产品设计 Review**：
- [ ] 页面结构合理
- [ ] 交互逻辑完整
- [ ] 组件状态覆盖完整
- [ ] 符合设计规范

**原型 Review**：
- [ ] 与设计一致
- [ ] 交互流畅
- [ ] 边界情况处理
- [ ] 性能良好

## 七、常见问题

**Q：PRD和产品设计有什么区别？**
A：PRD是"做什么"，产品设计是"怎么做"。PRD给需求提出人看，产品设计给UI/UX和开发看。

**Q：原型一定要HTML吗？**
A：HTML原型最接近最终效果，推荐使用。也接受Figma等设计稿，但需要有详细的交互说明。

**Q：Review不过怎么办？**
A：记录问题，修订后进入下一轮Review。通常2-3轮可以定稿。
