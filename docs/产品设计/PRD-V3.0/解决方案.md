#### 【V3-PT-01】客户转移默认值错误（自己转移给自己）

**优先级**：🔴 P1（上线阻塞）
**问题来源**：V2.0 第 2 轮测试发现，第 10 轮测试报告标记"已修复"，但 PM 验收报告仍列为 P1 遗留，需验证实际代码修复状态

**问题描述**：
批量转移弹窗中，目标经纪人下拉框的默认值仍为当前登录用户"陈佳佳"，即用户可以不改变默认值直接点击确认，将客户转移给自己。这不仅是一个体验问题，更是一个业务逻辑错误——自己不能成为自己的客户。

**根本原因分析**：
- 后端获取可转移经纪人列表时，未排除当前登录用户
- 前端弹窗初始化时，未主动设置一个非自己的默认值（通常应选择同店其他经纪人或清空让用户主动选择）

**解决方案**：

1. **后端修复**：
   - `GET /api/customers/transfer-targets` 接口，在返回可转移经纪人列表时，排除当前登录用户 (`current_user_id`)
   - 接口返回字段增加 `is_current_user` 标记，供前端判断是否需要特殊处理

2. **前端修复**：
   - 批量转移弹窗初始化时，检查默认选中值：
     - 如果默认选中项是当前用户，则自动选中列表中的第一个其他经纪人（或清空选择器并提示"请选择目标经纪人"）
     - 不允许在目标经纪人确认为自己的情况下触发提交
   - UI 层增加红色警示文案：「不能转移给自己，请重新选择目标经纪人」

3. **双重校验**：
   - 前端提交前再次校验 selected_target_id ≠ current_user_id
   - 后端接口增加防御性校验，拒绝 `target_agent_id == from_agent_id` 的请求

**验收标准**：
- [ ] 批量转移弹窗打开时，目标经纪人下拉框默认值不是当前用户
- [ ] 选择"自己"时，确认按钮仍然 disabled 或有明确警示
- [ ] 后端拒绝自己转给自己的请求，不报 500 而返回明确的错误码 `TRANSFER_TO_SELF`

**涉及文件**（预估）：
- 后端：CustomerService / transfer API
- 前端 PC：customer-list.html 批量转移弹窗逻辑

#### 【V3-PT-02】跟进保存空内容问题

**优先级**：🔴 P1（上线阻塞）
**问题来源**：V2.0 第 2 轮测试发现，PM 验收报告列为 P1-2 遗留

**问题描述**：
添加跟进记录弹窗中，用户不填写任何内容直接点击保存，系统提示"保存成功"并关闭弹窗，但实际入库的是一条空跟进记录。这不仅污染数据，还导致客户跟进时间线出现无意义的空节点。

**根本原因分析**：
- 前端表单提交时，未对跟进内容字段（follow_content）做非空校验
- 后端接口未拒绝空内容入库
- 关闭弹窗操作在保存回调之前，客户感知"保存成功"但实际可能失败

**解决方案**：

1. **前端新增表单校验**（三段式保护）：

   **第一段：提交前预校验**
   ```javascript
   // 跟进内容校验
   if (!followContent || followContent.trim().length === 0) {
     showToast('请填写跟进内容', 'error');
     return; // 不关闭弹窗，不触发保存
   }
   if (followContent.trim().length < 5) {
     showToast('跟进内容至少 5 个字符', 'error');
     return;
   }
   ```

   **第二段：禁用按钮防止重复提交**
   - 保存按钮点击后立即切换为 loading 状态（显示"保存中…"）
   - 禁用按钮并设置 3 秒超时，超时后自动恢复（防止请求卡死）

   **第三段：关闭弹窗的时机控制**
   - 只有当后端接口返回 `code: 200`（成功）时，才关闭弹窗
   - 接口返回失败时，弹窗保持打开，显示错误信息

2. **后端防御性校验**：
   ```python
   # 新增跟进记录接口
   def create_follow_up():
       content = request.json.get('content', '').strip()
       if not content or len(content) < 5:
           return jsonify({
               'code': 400,
               'message': '跟进内容不能为空且至少 5 个字符'
           }), 400
   ```

3. **数据库层约束**：
   - `follow_up_record` 表的 `content` 字段设置为 NOT NULL（应用层校验兜底）
   - 建议增加 content_length 字段存储实际字符长度，便于后续数据清洗

**验收标准**：
- [ ] 空内容提交时，弹窗不关闭，且显示"请填写跟进内容"提示
- [ ] 内容少于 5 字符提交时，弹窗不关闭，显示"跟进内容至少 5 个字符"
- [ ] 保存成功后弹窗关闭，且跟进列表实时刷新
- [ ] 后端接口对空内容返回 400 错误码，不入库

#### 【V3-NAV-02】PC 导航侧边栏 404（跨域加载失败）

**优先级**：🔴 P1（上线阻塞）
**问题来源**：UI-Review 第 5 轮遗留，PC 端侧边栏 404

**问题描述**：
PC 端页面加载时，左侧导航侧边栏显示 404 或内容空白。问题出现在侧边栏通过 `fetch()` 从后端动态加载菜单数据时，因跨域（CORS）问题导致请求失败。

**根本原因分析**：
- 前端页面通过 AJAX（fetch）从不同域名/端口的后端加载侧边栏配置
- 后端未配置 CORS 响应头（Access-Control-Allow-Origin）
- 前端未做跨域失败的降级处理，导致 404 展示

**解决方案**：

1. **后端 CORS 配置修复**（首选方案）：
   ```python
   # 后端全局 CORS 中间件
   app.add_middleware(
       CORSMiddleware,
       allow_origins=["http://localhost:3001", "https://pc.example.com"],
       allow_credentials=True,
       allow_methods=["GET", "POST", "OPTIONS"],
       allow_headers=["Content-Type", "Authorization"],
   )
   ```
   - 开发环境允许 localhost 端口
   - 生产环境配置实际域名白名单
   - 预检请求（OPTIONS）正确响应

2. **前端降级方案**（侧边栏数据加载失败时的兜底 UI）：
   ```javascript
   async function loadSidebar() {
     try {
       const res = await fetch('/api/sidebar/menu', {
         credentials: 'include'  // 携带 Cookie
       });
       if (!res.ok) throw new Error(`HTTP ${res.status}`);
       const data = await res.json();
       renderSidebar(data);
     } catch (err) {
       console.warn('侧边栏加载失败，使用默认配置:', err);
       renderSidebar(DEFAULT_SIDEBAR_MENU);  // 降级到本地默认配置
     }
   }
   ```

3. **本地默认菜单配置**（不依赖后端）：
   ```javascript
   const DEFAULT_SIDEBAR_MENU = [
     { id: 'building', text: '新盘管理', icon: 'building', children: [...] },
     { id: 'task', text: '任务管理', icon: 'task', children: [...] },
     { id: 'speech', text: '话术训练', icon: 'speech', children: [...] },
     { id: 'record', text: '录音分析', icon: 'record', children: [...] },
     { id: 'material', text: '物料管理', icon: 'material', children: [...] },
     { id: 'customer', text: '客户管理', icon: 'customer', children: [...] },
     { id: 'dashboard', text: '数据看板', icon: 'dashboard', children: [...] },
   ];
   ```

4. **HTTP 服务器配置**（如果使用 http-server 等静态服务器）：
   - 确保静态服务器配置了正确的 SPA fallback（所有路由回退到 index.html）
   - 避免 API 代理到后端时的路径问题

**验收标准**：
- [ ] PC 端所有页面左侧导航侧边栏正常展示，无 404
- [ ] 跨域请求正常返回数据（非降级默认菜单）
- [ ] 关闭后端服务时，侧边栏降级显示默认菜单（不白屏）

#### 2.1.2 动态字段变更的双端同步增强

**问题**：V2.0 虽然已设计 data_version + building_change_log，但实际同步机制还需明确触发时机和推送策略。

**同步策略**：

| 变更类型 | 触发动作 | 推送范围 | 小程序行为 |
|---------|---------|---------|-----------|
| 价格变更 | 保存后立即触发 | 该楼盘所有订阅者 | 详情页顶部显示「价格已更新」Banner，点击刷新 |
| 户型变更 | 保存后立即触发 | 该楼盘所有订阅者 | 列表页该楼盘卡片显示「户型有更新」角标 |
| 基本信息变更 | 保存后立即触发 | 该楼盘所有订阅者 | 详情页 data_version+1，重新拉取全量数据 |
| 配套信息变更 | 保存后延迟 5 分钟触发（批量） | 该楼盘所有订阅者 | T+5min 版本检测时自动刷新 |

**版本检测机制**（小程序端）：
```javascript
// 小程序新盘详情页 onShow 生命周期
onShow() {
  const version = wx.getStorageSync(`building_version_${this.buildingId}`);
  wx.request({
    url: `/api/building/${this.buildingId}/version`,
    success: (res) => {
      if (res.data.data_version > version) {
        this.refreshData(); // 版本号增加，重新拉取
        wx.setStorageSync(`building_version_${this.buildingId}`, res.data.data_version);
        wx.showToast({ title: '信息已更新', icon: 'none' });
      }
    }
  });
}
```

#### 2.3.1 训练数据回流的实时性保障

**问题**：V2.0 设计了数据回流机制，但未明确同步时机和失败重试策略。

**同步策略**：

| 训练完成时机 | 同步动作 | 超时处理 |
|------------|---------|---------|
| 单次训练答题完成 | 立即同步训练结果到后端 | 失败则本地缓存，网络恢复后重试（最多 3 次，间隔 30s） |

**小程序端离线缓存**：

```javascript
async function syncTrainingResult(result) {
  try {
    await wx.request({ url: '/api/training/result', method: 'POST', data: result });
  } catch (err) {
    // 存储到本地，待网络恢复后重试
    const pending = wx.getStorageSync('pending_training_sync') || [];
    pending.push({ result, timestamp: Date.now(), retryCount: 0 });
    wx.setStorageSync('pending_training_sync', pending);
  }
}

  if (res.isConnected) retryPendingSync();
});
```

#### 2.7.1 物料下载失败兜底方案

**问题**：V2.0 Review 第 5 轮遗留（#R5-02），物料下载失败的兜底方案未在 PRD 正文明确定义。

**兜底机制**：

| 失败场景 | 系统动作 | 用户感知 |
|---------|---------|---------|
| 网络中断 | 检测到下载中断 → 自动重试 3 次（间隔 5s）→ 均失败则提示 | 「下载失败，请检查网络后重试」+ 重试按钮 |
| 文件不存在（CDN 404） | 后端检测到文件缺失 → 标记为「素材已清理」+ 通知上传者 | 「该物料文件已清理，请联系上传者重新上传」 |
| 文件损坏 | MD5 校验失败 → 标记为「素材异常」 | 「文件损坏，请重新下载」或「联系管理员处理」 |
| 存储服务超时 | 3 次重试后标记「下载超时」 | 「下载服务繁忙，请稍后重试」 |

**下载重试逻辑**：
```javascript
async function downloadMaterial(materialId, fileUrl) {
  const maxRetries = 3;
  for (let i = 0; i < maxRetries; i++) {
    try {
      const res = await wx.downloadFile({ url: fileUrl });
      if (res.statusCode === 200) {
        await wx.saveImageToPhotosAlbum({ filePath: res.tempFilePath });
        wx.showToast({ title: '保存成功' });
        return;
      }
    } catch (err) {
      console.warn(`下载失败，第 ${i+1} 次重试:`, err);
      if (i === maxRetries - 1) {
        wx.showModal({
          title: '下载失败',
          content: '请检查网络后重试，如持续失败请联系管理员',
          confirmText: '重新下载',
          success: (res) => res.confirm && downloadMaterial(materialId, fileUrl)
        });
      }
    }
    await sleep(5000); // 等待 5s 后重试
  }
}
```

## 四、任务状态流转设计

> 核心问题：PC端和小程序端的任务状态名称、状态值、流转逻辑必须完全一致。本章节定义双端统一的状态机，作为开发和验收的基准。

### 4.1 统一状态字典

**探盘任务状态**（dict_task_status）：

| status_key | 统一展示文本（PC） | 统一展示文本（小程序） | 说明 |
|-----------|-----------------|---------------------|------|
| `pending` | 待执行 | 待执行 | 任务已下发，待开始 |
| `in_progress` | 进行中 | 进行中 | 正在执行，有阶段数据 |
| `paused` | 已暂停 | 已暂停 | 任务暂停（可恢复） |
| `submitted` | 已提交 | 已提交待审核 | 成果已提交，等待审核 |
| `approved` | 已通过 | 已通过 | 审核通过 |
| `rejected` | 已驳回 | 已驳回 | 审核驳回，需整改 |
| `expired` | 已超时 | 已超时 | 超过截止时间未完成 |

**任务阶段状态**（task_checkpoint_status）：

| status_key | 展示文本 | 说明 |
|-----------|---------|------|
| `locked` | 未解锁 | 前置阶段未完成 |
| `active` | 进行中 | 当前阶段，可执行 |
| `completed` | 已完成 | 该阶段数据已提交 |
| `skipped` | 已跳过 | 配置为跳过（部分阶段可跳过） |

### 4.2 状态流转图

**探盘任务主状态流转**：

```
                    ┌──────────────────────────────────────────────────┐
                    │                                                  │
                    ↓                                                  │
pending ──→ in_progress ──→ submitted ──→ approved                        │
                    │              │                                    │
                    │              ├────────→ rejected ──→ submitted     │
                    │              │           ↑                       │
                    │              │           │                       │
                    │              │      (整改后重新提交)               │
                    │              │                                    │
                    └──→ expired ←─┘                                    │
                         ↑                                              │
                    (逾期未提交)                                        │
```

**子阶段流转**（阶段1→阶段2→阶段3→阶段4）：

```
locked ──→ active ──→ completed
           │
           └──(该任务被驳回)──→ locked（从当前阶段重新开始）
```

### 4.3 状态流转触发条件

| 当前状态 | 触发事件 | 目标状态 | 触发方 | 两端同步方式 |
|---------|---------|---------|-------|------------|
| pending | 任务到达开始时间 / 经纪人点击"开始" | in_progress | 小程序端 | 实时同步 |
| in_progress | 所有阶段 completed | submitted | 小程序端 | 实时同步 |
| submitted | PC端审核通过 | approved | PC端 | WebSocket + 轮询兜底 |
| submitted | PC端审核驳回 | rejected | PC端 | WebSocket + 轮询兜底 |
| rejected | 小程序端整改完成重新提交 | submitted | 小程序端 | 实时同步 |
| in_progress | 超过截止时间且无打卡数据 | expired | 后端定时任务 | 消息队列 |
| paused | PC端/小程序端主动暂停 | paused | 双端 | 实时同步 |
| paused | 小程序端点击"恢复" | in_progress | 小程序端 | 实时同步 |

### 4.4 双端状态同步机制

**实时同步方案**（推荐）：

```javascript
// PC端：WebSocket 监听任务状态变更
const ws = new WebSocket('wss://api.example.com/ws/task-status');
ws.onmessage = (event) => {
  const { task_id, new_status, changed_at } = JSON.parse(event.data);
  updateTaskStatusUI(task_id, new_status);
  // 如果当前页面正在查看被更新的任务详情，刷新详情页
  if (currentTaskId === task_id) {
    refreshTaskDetail(task_id);
  }
};

// 小程序端：使用 wx.connectSocket（与 PC 端相同的 ws 地址）
wx.connectSocket({
  url: 'wss://api.example.com/ws/task-status',
});

// 后端：任务状态变更时广播
function updateTaskStatus(taskId, newStatus) {
  db.update('tasks', { status: newStatus, updated_at: now() }, { id: taskId });
  // 广播到所有订阅者
  wsServer.broadcast('task_status_change', {
    task_id: taskId,
    new_status: newStatus,
    changed_at: now()
  });
  // 发送小程序服务通知（如果 PC 端操作导致变更）
  if (isPCOperator) {
    sendMiniAppNotification(taskId, newStatus);
  }
}
```

**轮询兜底方案**（WebSocket 不可用时）：

| 场景 | 轮询策略 |
|------|---------|
| PC端任务列表页 | 每 30s 自动刷新任务状态（仅当页面可见时）|
| PC端任务详情页 | 每 10s 轮询状态（页面失焦时停止）|
| 小程序端任务列表 | 下拉刷新（用户主动触发），无自动轮询 |
| 小程序端任务详情 | 进入页面时检测 data_version 是否变化 |

### 4.5 各模块状态机汇总

| 模块 | 状态数量 | 关键状态 | 状态字典表 |
|------|---------|---------|-----------|
| 探盘任务 | 7个（主）+ 4个/阶段 | approved/rejected/expired | dict_task_status |

| 录音分析 | 4个（uploading/processing/completed/failed） | completed/failed | dict_audio_analysis_status |
| 物料 | 3个（draft/published/archived） | published | dict_material_status |
| 客户 | 无主状态，有操作日志 | — | — |
