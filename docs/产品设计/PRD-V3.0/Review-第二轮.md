## 三、任务状态流转专项验证

> 重点：第一轮发现任务状态流转是最大风险项，需重点验证当前PRD是否已修正

### 3.1 双端状态key对比

| 维度 | Solution定义 | PRD定义（5.8.4数据结构） | PRD定义（5.9.5状态说明） | 一致性 |
|------|------------|------------------------|------------------------|--------|
| 数据结构中的status | 英文key | ✅ pending/progress/completed/timeout | ❌ 中文"待分发"等 | ⚠️ 部分一致 |
| 状态说明文本 | 英文key→中文展示 | ❌ 无状态说明 | ❌ 仅中文名称 | ⚠️ 需对齐 |

### 3.2 状态key不一致问题

**第5.8.4.1节**（探盘任务数据结构）：
```javascript
status: "progress"  // pending/progress/completed/timeout
```
→ 使用英文status key

**第5.9.5.1节**（任务状态说明）：
```markdown
| 状态 | 说明 | 小程序端可见操作 |
|------|------|-----------------|
| 待分发 | 创建后未下发 | 下发、编辑、删除 |
```
→ 使用中文状态名称

**问题**：同一个PRD中，两处对任务的status定义使用不同的命名体系。Solution要求统一使用英文status_key作为内部标识，PRD需要补充状态字典表进行统一。

### 3.3 状态流转图对比

**Solution流转**：
```
pending → in_progress → submitted → approved/rejected → expired
                   ↑                  ↓
              paused ←────────── rejected（整改后）
```

**PRD流转**（第1697行）：
```
待分发 → 待执行 → 进行中 → 已提交 → 已完成/已驳回
                          ↘ 已逾期（可继续执行）
```

**关键差异**：
1. Solution有paused（暂停），PRD无
2. Solution无"待分发"状态，PRD有
3. Solution rejected后→submitted（重新提交），PRD是"重新下发"
