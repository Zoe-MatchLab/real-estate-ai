# PRD工作流技能

> **版本**: V2.0
> **日期**: 2026-04-13
> **用途**: 端到端的PRD文档工作流程，包含需求→设计→原型→评审

---

## 1. 技能概述

### 1.1 技能定位
本技能用于指导**PRD需求文档**的完整工作流程，让需求提出人能够直观看到需求的实现效果。

### 1.2 核心价值
- **需求提出人**: 通过PRD+原型直观看到最终效果
- **UI/UX设计师**: 依据PRD输出产品设计
- **FE开发**: 基于原型直接开发
- **测试**: 依据PRD和原型编写测试用例，提前参与Review确保质量

### 1.3 触发场景
- 创建新模块的PRD
- Review PRD文档
- 梳理需求文档结构
- 建立标准化的需求管理流程

  需求提出人         PM              UI/UX          FE开发          测试
      │              │                │               │               │
      ▼              ▼                ▼               ▼               ▼
┌─────────┐   ┌─────────┐   ┌─────────┐   ┌─────────┐   ┌─────────┐
│0.聊需求 │   │1.PRD撰写│   │2.产品设计│  │3.原型实现│  │4.测试准备│

              ┌─────────────────────────────────────────────────────┐
              │                   Review（多端并行）                    │
              │  PRD Review → 设计 Review → 原型 Review → 测试确认   │
              └─────────────────────────────────────────────────────┘
                                    │
                                    ▼
                              ┌─────────┐
                              │ 5.定稿  │
                              └─────────┘
```

### 3.2 流程说明

> **【重要】第一步：聊清楚需求！** 需求不聊清楚，后续所有工作都是白费。

| 阶段 | 角色 | 输出物 | 说明 |
|-----|-----|-------|-----|
| 0. 聊清楚需求 | 需求提出人 + PM | 需求纪要 | **第一优先级！**把需求聊清楚 |
| 1. 需求确认 | PM + 需求提出人 | 确认清单 | 端确认、主题色确认 |
| 2. PRD撰写 | PM | PRD文档 | 按端独立撰写 |
| 3. 产品设计 | UI/UX | 产品设计文档 | 按端独立设计、输出组件清单 |
| 4. 原型实现 | FE | HTML原型 | 按端实现、包含Mock数据 |
| 5. 测试准备 | 测试 | 测试用例 | 基于PRD和原型编写 |
| 6. Review | 所有人 | Review记录 | 各端并行Review |
| 7. 定稿 | PM | PRD终稿 | 合并所有输出物 |

## 7. Review流程

### 7.1 多端独立Review

```
需求纪要确认
     │
     ▼
PRD Review（PM + 需求提出人 + 测试）
     │
     ▼
┌─────────────────────────────────────────────────────┐
│              多端并行 Design Review                   │
├─────────────────────────────────────────────────────┤
│                                                          │
│  PC端设计 Review      小程序端 Review       APP端 Review   │
│       │                    │                   │         │
│       ▼                    ▼                   ▼         │
│  PC原型 Review       小程序原型 Review    APP原型 Review    │
│       │                    │                   │         │
│       ▼                    ▼                   ▼         │
│  [PC-RoundN]          [小程序-RoundN]    [APP-RoundN]    │
│       │                    │                   │         │
└───────┼────────────────────┼───────────────────┼─────────┘
        │                    │                   │
        └────────────────────┼───────────────────┘
                           │
                    所有端Review通过
                           │
                           ▼
                        定稿
```

### 7.2 Review检查清单

**PRD Review**：
- [ ] 需求背景清晰
- [ ] 用户故事完整，每个P0至少有1条量化验收标准
- [ ] 异常/边界场景已识别并记录
- [ ] 各端差异点已明确
- [ ] 性能指标已量化
- [ ] 测试关注点已识别

**产品设计 Review**：
- [ ] 页面结构合理
- [ ] 组件清单完整
- [ ] 组件状态覆盖完整（正常/异常/空/加载/禁用）
- [ ] 符合设计规范
- [ ] 主题色与确认清单一致
- [ ] Mock数据结构与页面字段一致

**原型 Review**：
- [ ] 与设计一致
- [ ] 交互流程完整
- [ ] 边界情况有处理
- [ ] 性能可接受（加载<2秒）
- [ ] 所有组件状态可演示

**测试确认**：
- [ ] 验收标准可转化为测试用例
- [ ] 边界条件已覆盖
- [ ] 兼容性要求已明确

## 10. 常见问题

**Q: PRD和产品设计有什么区别？**
A: PRD是"做什么"，产品设计是"怎么做"。PRD给需求提出人看，产品设计给UI/UX和开发看。

**Q: 原型一定要HTML吗？**
A: HTML原型最接近最终效果，推荐使用。也接受Figma等设计稿，但需要有详细的交互说明和Mock数据。

**Q: 多端原型如何管理？**
A: 各端独立Review，PRD中汇总链接。命名保持一致便于关联。

**Q: Review不过怎么办？**
A: 记录问题，修订后进入下一轮Review。最多3轮，超时由PM升级决策。
