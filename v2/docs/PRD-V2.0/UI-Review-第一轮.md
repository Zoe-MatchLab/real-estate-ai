## 二、交互可用性检查

### 2.1 按钮与表单

| 页面 | 问题 |
|------|------|
| 所有PC页面 | 表单 `input`/`select` 有 `:focus` 状态（`focus:ring-primary/30`），但**无实时验证反馈**（如必填项为空时的错误提示） |
| `customer-detail.html` | 跟进方式按钮（电话/微信/面谈）**未绑定切换事件**，视觉上仅 CSS `:active` 样式，无 JS 交互 |
| `customer-list.html` | 批量转移按钮触发 `openTransferModal()`，但 Modal 内的下拉选择后**无校验**，可空提交 |
| `record-upload.html`（小程序） | 文件上传后**无上传进度条**，无上传结果状态反馈（成功/失败/文件大小超限） |

### 2.2 弹窗/抽屉

| 页面 | 问题 |
|------|------|
| 所有 Modal | 弹出后**背景内容仍可滚动**（无 `overflow: hidden`），建议在 `body` 加 `modal-open` 类控制 |
| `task-audit.html` 驳回弹窗 | 结构化驳回设计良好，分类选择清晰；但驳回原因**可多选却设计为单选交互**（点击另一项自动取消上一项），需求是否支持多选需确认 |
| `followModal` | 跟进方式按钮组切换逻辑未实现，当前仅样式切换 |
| 弹窗关闭方式 | 均支持点击 X 按钮关闭，但**点击遮罩层不能关闭**（需手动点取消按钮），建议增加 `onclick="close..."` 在遮罩层 |

### 2.3 导航与链接

- ❌ **侧边栏组件未内置**：`pc/pages/v2/` 下所有页面均通过 `fetch('../../components/sidebar.component.html')` 异步加载侧边栏，若组件路径不正确或网络异常，页面将**侧边栏区域空白**，无降级处理
- ⚠️ **返回链接**：小程序端 `record-upload.html` 导航栏有返回按钮，但部分页面返回链接未指定实际跳转路径
- ✅ 面包屑导航在 PC 端各页面结构完整

### 2.4 分页与批量操作

- ⚠️ `customer-list.html`（PC）：全选 checkbox 有 `toggleSelectAll()`，但**选择后未实时显示"已选 N 项"**，批量操作区无感知反馈
- ⚠️ `task-list.html`、`record-list.html`（PC）：分页组件**仅有静态 UI，无翻页交互逻辑**

## 四、响应式适配检查

### 4.1 PC端响应式 ⚠️ 未覆盖

- 所有 PC 页面使用固定布局（侧边栏 `w-64 fixed`），**无媒体查询** `@media (max-width: ...)`

具体风险：
- 1280px 以下：侧边栏可能遮挡内容区
- 1024px（iPad 横屏）：表格列可能溢出
- 768px 以下：统计卡片 grid (`grid-cols-5`) 会堆叠异常

建议补充断点：
```css
/* ≥1280px: 正常布局 */
/* 1024-1279px: 侧边栏收窄 */
/* <1024px: 侧边栏变为顶部导航或汉堡菜单 */
```

### 4.2 小程序端适配 ✅ 基本良好

- `maximum-scale=1.0, user-scalable=no` 锁定缩放，基础安全
- 横向 Tab 使用 `overflow-x: auto` + `scrollbar-width: none` 防止滚动条出现
- 底部固定栏（Tab Bar）固定 `position: fixed`，不受内容区滚动影响
- ⚠️ 未测试 iPhone SE（小屏 320px）及大屏平板（768px+）的实际渲染效果

### 4.3 无障碍（a11y）⚠️ 未覆盖

- 所有图标按钮（`i.fa-xxx`）均无 `aria-label`，屏幕阅读器无法识别
- 表单 `input`/`select` 均无关联 `label` 元素（用 `placeholder` 代替），不符合无障碍规范
- 颜色对比度：部分灰色文字（如 `#86909C` 在白底上）对比度偏低，建议降级为深色辅助文字

## 六、建议

### 6.1 优先级处理顺序

```
P0 → P1 → P2
```

建议修复顺序：
1. **侧边栏内置**（P0-1）：将 sidebar HTML 直接嵌入各页面或提供本地 fallback
2. **进度条逻辑修正**（P0-2）：与产品确认"提交审核"是否为独立阶段
3. **上传反馈**（P0-3）：增加上传进度条和结果状态
4. **表单验证**（P1-1）：增加实时验证和内联错误提示
5. **Modal 体验优化**（P1-3~P1-6）
6. **跨端配色统一**（P1-7）：明确两端是否共用同一品牌色规范

### 6.2 设计规范建立建议

- **建立跨端 Design Token 文档**：定义品牌色、语义色、间距、圆角、阴影的系统化命名，供 PC 和小程序两端引用
- **组件库化**：将侧边栏、顶部导航、统计卡片、标签、Modal、表格等通用组件抽离，避免每个页面独立编写
- **移动端优先重构**：建议 `speech-data.html` 训练数据中心采用响应式重构，适配 iPad 场景

### 6.3 下一轮 Review 重点

- P0 问题修复后的回归验证
- 各页面核心用户流程（创建任务→执行→提交→审核→通过）的端到端串联测试
- 特殊状态：加载中、空状态、错误状态的设计完整性
- 无障碍合规（WCAG 2.1 AA 标准）

## 七、结论

**整体评价：原型基础框架完整，视觉风格相对统一，V2.0 新增功能（结构化驳回、过程数据Tab、批量客户转移、物料中心）设计思路清晰。**

**主要风险集中在三个方面：**

1. **侧边栏外部依赖**是最高风险点，可能导致生产环境页面布局整体失效
2. **表单和弹窗交互缺失**会影响审核等核心业务流程的可用性
3. **两端品牌色不统一**将影响用户的跨端一致品牌认知

建议在下一轮迭代前完成所有 P0 问题的修复，P1 问题建议在 V2.0 正式发布前处理，P2 问题可排入后续优化版本。
