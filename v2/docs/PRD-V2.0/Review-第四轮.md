## 一、Review概述

### 1.1 本轮验证目标

前三轮Review已覆盖：逻辑完整性、数据流向、接口字段、错误处理、安全权限。本轮作为第四轮，聚焦**API契约文档编写视角**，验证PRD中所有接口的可实现性，确保开发团队能够基于PRD直接输出API接口文档，无需再做额外的产品澄清。

### 1.2 本轮验证框架

| 验证维度 | 核心问题 | 验证方法 |
|----------|----------|----------|
| **接口可实现性** | 请求/响应字段是否明确？数据类型是否定义？边界值是否有说明？ | 逐模块审查接口字段表，对照解决方案的数据字段变更 |
| **与现有系统的兼容性** | 新功能是否需要改造现有系统？数据迁移方案是否有？ | 识别新增表 vs 改造表，评估迁移复杂度 |
| **性能考量** | 大数据量场景是否有优化方案？缓存策略是否合理？ | 检查分页/排序/缓存/CDN等性能相关设计 |
| **依赖关系** | 新功能之间的依赖是否清晰？外部系统依赖是否有说明？降级策略是否有？ | 梳理功能依赖图和外部依赖清单 |

### 1.3 遗留问题承接

| 轮次 | 阻塞问题 | 建议问题 | 关键遗留 |
|------|----------|----------|----------|
| 第一轮 | 2 | 18 | 7个功能缺失（PC端录音分析/客户管理/训练数据中心等） |

#### 2.1.2 打卡数据同步接口

**当前PRD描述（解决方案#DP-01）：**

```javascript
// 每次打卡提交触发同步
POST /api/checkpoint/sync
{
  taskId: "TASK_xxx",
  checkpointId: 1,
  gpsLat: 31.22,    // GPS纬度
  gpsLng: 121.55,   // GPS经度
  gpsAddress: "上海市浦东新区...",  // 地址解析
  photoUrls: ["https://..."],  // 照片URL数组
  captureTimestamp: "2026-03-31T10:30:00Z",
  sequenceOrder: 1   // 打卡顺序
}
```

**评估结论：⚠️ 可实现但需补充3处**

| 问题 | 说明 | 建议 |
|------|------|------|
| #API-PT-01 | `photoUrls` 为URL数组，但未定义单个文件大小限制 | 补充：`photoUrls[].size ≤ 10MB`，格式jpg/png/webp |
| #API-PT-02 | 未定义照片数量上限 | 建议：`photoUrls.length ≤ 20`，视频 `videoUrls.length ≤ 5` |
| #API-PT-03 | 未定义GPS坐标精度要求 | 建议：精度≤6位小数，超出自动截断 |

| #API-ST-02 | 录音练习记录（`training_audio` 表）未定义上传接口 | 补充：录音练习同步接口，字段包含 `audio_url`, `audio_duration`, `ai_score` |
| #API-ST-03 | 重复训练校验规则（24小时间隔）在API层如何校验？ | 建议：后端新增 `training_record` 表增加 `is_valid_record` 布尔字段，同步时由后端判断是否记为有效记录 |

## 三、与现有系统的兼容性验证

### 3.1 新增表 vs 改造表现状

| 表名 | 操作类型 | 改造影响范围 | 数据迁移需求 |
|------|----------|-------------|-------------|
| `task_record`（打卡记录） | 新增 | 小程序端打卡逻辑改造 | 无（全新数据） |
| `training_record`（训练记录） | 新增 | 小程序端训练逻辑改造 | 无（全新数据） |
| `call_analysis_result`（录音分析） | 新增 | 小程序端录音上传逻辑改造 | 无（全新数据） |
| `customers`（客户表） | 新增 | 小程序端客户管理逻辑改造 | 无（全新数据） |
| `follow_up_record`（跟进记录） | 新增 | 小程序端跟进逻辑改造 | 无（全新数据） |
| `materials`（物料表） | 改造 | 现有探盘成果汇聚逻辑改造 | 历史物料迁移方案缺失 |
| `buildings`（楼盘表） | 改造 | 现有新盘管理逻辑改造 | 需补充历史数据补充 `tag_ids` 和 `data_version` 字段 |
| `question_banks`（题库表） | 新增 | 全新模块 | 无 |
| `question_bank_tags`（题库标签） | 新增 | 全新模块 | 无 |

**评估结论：⚠️ 存在2处兼容性风险**

| 风险 | 说明 | 影响 |
|------|------|------|
| #COM-01 | `buildings` 表新增 `tag_ids[]` 和 `data_version` 字段，历史楼盘数据如何补充？ | 所有存量楼盘 `tag_ids` 为空，`data_version` 默认为1，不影响功能但可能导致题库匹配降级为通用题库 |
| #COM-02 | `materials` 表新增 `publish_status`、`visibility_scope`、`thumbnail_url` 字段，历史物料如何迁移？ | 建议：历史物料默认 `publish_status=published`、`visibility_scope=all`，避免存量物料不可见 |
| #COM-03 | 新增字段的默认值在数据库层面需要明确 | 建议：所有新增字段在DDL中明确 DEFAULT 值，避免NULL判断导致业务逻辑复杂化 |

### 3.2 外部系统依赖兼容性

| 外部系统 | 依赖接口/功能 | 接入方式 | 兼容性风险 |
|----------|-------------|----------|-----------|
| 微信小程序 | 录音文件上传、订阅消息推送 | 微信小程序API | ⚠️ 微信政策变更可能导致录音上传方式受限 |
| AI大模型 | 内容生成（文案/封面图/视频）、话术分析 | 第三方API（OpenAI/国内大模型） | 🔴 高风险：AI模型切换成本高，Prompt耦合严重 |
| 腾讯地图 | GPS定位打卡 | 腾讯地图WebService API | ✅ 低风险，地图服务稳定 |
| 企业微信 | 组织架构同步 | 企业微信API | ✅ 低风险，已有成熟方案 |
| 云存储（对象存储） | 录音/图片/视频存储 | 第三方OSS（如阿里云OSS） | ⚠️ 需提前定义存储分层策略（热/冷数据） |

**评估结论：⚠️ AI大模型依赖存在高风险**

| 问题 | 说明 | 建议 |
|------|------|------|
| #COM-AI-01 | AI生成Prompt与业务逻辑高度耦合，Prompt变更可能影响生成质量 | 建立Prompt版本管理，每套Prompt变更记录版本号 |
| #COM-AI-02 | 无AI降级方案（如大模型服务不可用时的兜底） | 建议：定义降级策略，如"文案生成失败时展示上次生成结果+人工标注需重试" |
| #COM-AI-03 | AI模型切换（如从GPT切换到国内模型）成本评估缺失 | 建议：预留Prompt适配工时（每个AI功能1-2人天） |

### 4.2 缓存策略

## 五、依赖关系验证

### 5.1 新功能间依赖关系

**功能依赖图：**

```
[新盘上线] 
    ↓ 触发
[自动关联训练题库] ←→ [题库-楼盘关联配置]（需先配置）
    ↓ 触发
[下发探盘任务]
    ↓ 并行
├── [探盘打卡] → [点位数据同步] → [PC端审核辅助]
├── [话术训练] → [训练数据同步] → [PC端训练数据中心]
└── [录音分析] → [分析结果同步] → [PC端录音分析管理]
    ↓
[物料库汇聚] → [物料发布] → [小程序物料中心]
    ↓
[数据看板]（依赖上述所有数据）
```

**评估结论：⚠️ 存在2处依赖风险**

| 问题 | 说明 | 风险级别 |
|------|------|----------|
| #DEP-01 | 数据看板依赖上游所有功能数据，上游功能延期会直接影响看板上线 | 🔴 高风险：建议将看板与上游功能解耦，上游功能分期上线 |
| #DEP-02 | 小程序物料中心依赖PC端物料发布流程，若PC端发布功能延期，小程序物料中心无法展示内容 | 🟡 中风险：建议明确物料发布流程的优先级 |
| #DEP-03 | PC端审核辅助工具（#DP-01）依赖小程序打卡数据实时同步，若同步机制延期，PC端审核功能无法使用 | 🟡 中风险：建议同步机制作为独立Story提前开发 |

### 5.3 降级策略

**当前PRD降级策略覆盖情况：**

| 场景 | 降级策略 | 评估 |
|------|----------|------|
| AI内容生成失败 | 无明确定义（#EC-02第三轮遗留） | ❌ 缺失 |

| 打卡GPS定位失败 | 无兜底方案（#EC-02第三轮遗留） | ❌ 缺失 |
| 小程序后台生成过程中断 | "后台服务持续运行" | ✅ 有明确定义 |

**评估结论：❌ 降级策略体系未建立**

| 问题 | 说明 | 建议 |
|------|------|------|
| #DEGRADE-01 | AI生成失败无降级策略 | 定义生成任务状态机：`pending→generating→completed/failed`，失败自动重试2次，仍失败通知用户并保留手动重试 |
| #DEGRADE-02 | AI分析超时无告警机制 | 定义：分析超过24小时自动告警到运维，同时通知用户预计延迟 |
| #DEGRADE-03 | 打卡GPS定位失败无兜底 | 定义：GPS失败时允许"手动定位"（用户选择楼盘坐标作为打卡位置），需记录并标记"人工确认" |
| #DEGRADE-04 | 重试机制策略未标准化（#EC-03第三轮遗留） | 统一重试策略：最多3次，间隔1s/5s/30s（指数退避），超限记录失败+提示手动重试 |
