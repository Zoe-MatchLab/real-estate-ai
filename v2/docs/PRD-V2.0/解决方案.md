### 【#PT-01】审核驳回原因未结构化传递到小程序端

**优先级**：🔴高
**所属模块**：探盘任务
**问题描述**：PC端审核人员填写驳回原因时，目前是自由文本方式。小程序端经纪人只能看到一条文字描述，无法结构化展示驳回原因的类型、具体点位、缺少的材料等关键信息，导致经纪人看到驳回后仍不清楚如何修正。

**解决方案**：

1. **定义驳回原因结构化模板**：
   - 建立标准驳回原因分类（缺少材料、信息错误、位置偏差、未按要求拍摄、质量不达标、其他），每类下设二级原因和填写提示
   - PC端审核界面改为表单化驳回：先选驳回大类（必选），再选具体原因（必选），最后可补充自由说明（非必填，最长200字）

2. **小程序端驳回详情页重构**：
   - 将结构化驳回信息渲染为「驳回原因类型 + 要点列表 + 补充说明」三段式展示
   - 增加「查看整改示例」入口（链接到正确操作示例文档或视频），降低理解成本
   - 驳回记录支持列表展示，同一任务多次驳回的历史记录均可见

3. **驳回通知增强**：
   - 小程序收到驳回后，首页任务卡片显示红点角标，并推送服务通知或模板消息
   - 通知内容包含：任务名称 + 驳回类型 + 「立即整改」快捷按钮

**涉及的数据字段变更**：
- 新增字段：`rejection_reason_type`（驳回原因大类，枚举：missing_material / wrong_info / location_deviation / quality_fail / other）
- 新增字段：`rejection_reason_detail`（驳回原因明细，枚举数组，依赖类型字段）
- 新增字段：`rejection_reference_link`（整改参考链接，非必填）
- 修改字段：`rejection_free_text` → 保留为补充说明，字数限制200字
- 修改字段：`rejection_record` 结构：原单一文本 → 数组，每条记录包含上述结构化字段 + 审核人 + 驳回时间

**涉及的流程变更**：
- 新增流程：驳回原因结构化填写流程（PC端）
- 修改流程：驳回通知推送流程（新增小程序通知链路）

**两端影响**：
- **PC端**：审核驳回界面从文本框改造为「分类选择 + 明细选择 + 补充文本」三段式表单；驳回历史记录查询支持
- **小程序端**：任务详情页驳回区块改为结构化展示；新增首页红点提醒和快捷整改入口；驳回历史列表页

**优先级建议**：P0（必须）
**预计工时**：5人天（PC端表单改造2天 + 小程序端展示改造2天 + 联调1天）

### 【#DP-01】打卡点位完成记录仅小程序可见

**优先级**：🔴高
**所属模块**：探盘任务
**问题描述**：经纪人在小程序端完成打卡时，系统会记录每个点位的GPS坐标、拍摄时间、照片等过程数据。但PC端审核页面完全没有展示这些数据的入口，审核人员只能看到最终提交结果，无法核验点位真实性、拍摄顺序合理性，审核形同「盲审」，存在巨大管控漏洞。

**解决方案**：

1. **PC端审核页新增「过程数据」Tab页**：
   - 展示内容：各点位打卡时间序列、GPS坐标及地址解析、打卡照片（缩略图可放大）、点位完成顺序、「异常」标识（同一位置重复打卡、GPS严重偏移、快速连续拍摄等场景）
   - 异常检测规则预置（见下方数据字段变更）

2. **定义过程数据标准存储结构**：
   - 将小程序端打卡过程数据（点位维度）同步到后端任务明细表，不再仅存储在小程序本地
   - 审核人员可查看每个点位的「打卡质量评分」（系统根据GPS偏离度、拍摄间隔等自动计算，0-100分）

3. **PC端审核辅助工具**：
   - 地图打点展示：基于各点位GPS坐标，在地图上还原经纪人行进轨迹，便于判断真实性
   - 「异常」一键标记：审核人员可对可疑点位打标记，标记内容计入审核记录

**涉及的数据字段变更**：
- 新增字段：`checkpoint_records`（打卡点位记录数组，每条包含：`checkpoint_id`、`checkpoint_name`、`gps_lat`、`gps_lng`、`gps_address`、`photo_urls[]`、`capture_timestamp`、`sequence_order`、`quality_score`、`is_anomaly`、`anomaly_reason`）
- 新增字段：`task_trace_map_url`（轨迹图快照URL，非必填）
- 新增字段：`gps_deviation_score`（GPS偏离度评分，数值0-100）
- 修改字段：`task_record` 表增加 `has_checkpoint_data`（布尔）字段用于列表筛选

**涉及的流程变更**：
- 新增流程：小程序端打卡数据实时同步后端流程（每次打卡提交即触发同步）
- 修改流程：PC审核页面加载流程（新增获取打卡过程数据接口调用）

**两端影响**：
- **PC端**：审核详情页新增「过程数据」Tab（包含时间轴、地图轨迹、照片墙、异常提示）；审核列表支持「过程数据缺失」筛选标签
- **小程序端**：打卡数据提交逻辑改造，增加提交后立即同步后端的API调用；网络异常时本地缓存，网络恢复后自动重试同步

**优先级建议**：P0（必须）
**预计工时**：8人天（小程序数据同步改造2天 + PC端过程数据Tab开发3天 + 地图轨迹组件2天 + 联调1天）

### 【#DP-02】驳回原因字段未结构化定义

**优先级**：🔴高
**所属模块**：探盘任务
**问题描述**：与#PT-01关联但更底层——当前数据库层面 `rejection_reason` 字段定义为 TEXT 自由文本，未建立结构化枚举和校验规则。这导致PC端驳回表单无法做输入约束，小程序端展示逻辑也只能做模糊匹配，整体数据质量无法保障。

**解决方案**：

1. **数据库字段改造**：
   - 废弃现有的 `rejection_reason` TEXT字段
   - 新建结构化字段组：`rejection_type`（一级枚举）、`rejection_detail_code`（二级枚举）、`rejection_supplement`（补充文本，非必填）

2. **建立驳回原因枚举字典表**（`dict_rejection_reason`）：
   ```
   | type_code | type_name   | detail_codes[]          | detail_names[]                    |
   |-----------|-------------|--------------------------|------------------------------------|
   | RM        | 缺少材料     | RM_01, RM_02, RM_03       | 缺少身份证/缺少工牌/缺少楼盘介绍   |
   | WI        | 信息错误     | WI_01, WI_02, WI_03       | 楼盘名称错/价格信息错/户型信息错    |
   | LD        | 位置偏差     | LD_01, LD_02               | GPS严重偏移/未到规定位置            |
   | QF        | 质量不达标   | QF_01, QF_02, QF_03        | 照片模糊/光线不足/角度不符合要求     |
   | OT        | 其他         | OT_01                      | 其他原因                            |
   ```
   - 字典表支持后台管理可配置

3. **接口层数据校验**：
   - 驳回原因提交接口增加枚举值白名单校验，`rejection_detail_code` 必须属于 `rejection_type` 的合法子项
   - 数据回传时同时返回枚举值和展示文本，前端不做翻译逻辑

**涉及的数据字段变更**：
- 新增字段：`dict_rejection_reason` 字典表（type_code, type_name, detail_codes, detail_names, sort_order, is_active）
- 新增字段：`rejection_type`（驳回类型枚举，非空）
- 新增字段：`rejection_detail_code`（驳回明细代码，非空，依赖类型）
- 修改字段：`rejection_reason` TEXT → 保留作为 `rejection_supplement` 字段（限制200字，非必填）
- 新增索引：(`rejection_type`, `rejection_detail_code`) 联合索引，支撑统计报表

**涉及的流程变更**：
- 新增流程：驳回原因枚举字典管理流程（后台可配置）
- 修改流程：驳回提交接口数据校验流程（枚举值强校验）

**两端影响**：
- **PC端**：驳回表单UI适配新的枚举结构；后台管理增加驳回原因字典配置页
- **小程序端**：驳回展示组件适配新的字段结构；枚举值与展示文本分离处理

**优先级建议**：P0（必须）
**预计工时**：4人天（字典表设计与创建1天 + 接口校验改造1天 + PC端UI适配1天 + 小程序端适配1天）

### 【#ST-01】训练数据无法回流到PC端

**优先级**：🔴高
**所属模块**：话术训练
**问题描述**：经纪人在小程序端完成话术训练（答题、录音练习、AI评分）后，产生的训练记录（成绩、错题、录音）完全存储在小程序本地或小程序云，不存在PC端查看入口。区域经理无法在PC端查看团队经纪人的训练数据，训练模块对管理层完全闭环断裂。

**解决方案**：

1. **建立训练数据同步机制**：
   - 每次训练完成后，小程序端自动将训练结果同步至后端API（包含训练时间、楼盘ID、题型、得分、错题记录、录音文件Key）
   - 后端存储到 `training_record` 表，支持按经纪人、时间段、楼盘、题型等多维度查询
   - 录音文件上传至云存储，将返回的文件Key和URL一并存入记录

2. **PC端训练数据中心**：
   - 新增「训练数据中心」一级菜单（或挂在经纪人管理下作为Tab）
   - 展示内容：训练完成次数、平均得分、错题分布、录音练习数量等汇总
   - 支持 drill-down：点击汇总指标 → 跳转至经纪人粒度的训练明细列表
   - 支持按时间范围、楼盘、经纪人筛选

3. **数据权限控制**：
   - 区域经理仅可见自己管辖范围内经纪人的训练数据
   - 系统管理员可见全部

**涉及的数据字段变更**：
- 新增字段：`training_record` 表新增字段 `sync_status`（已同步/同步失败）、`sync_timestamp`
- 新增字段：`training_record` 表新增 `building_id`（关联楼盘ID）、`question_id`（关联题目ID）
- 新增字段：`training_audio` 表（录音练习记录表：record_id, agent_id, building_id, audio_url, audio_duration, ai_score, sync_status, created_at）
- 新增字段：`wrong_answer_log` 表（错题记录表：log_id, agent_id, training_record_id, question_id, user_answer, correct_answer, created_at）

**涉及的流程变更**：
- 新增流程：小程序训练完成 → 立即同步后端API → 后端写入数据库
- 新增流程：PC端查询训练数据流程（区域经理筛选 → 拉取管辖经纪人列表 → 查询训练记录）

**两端影响**：
- **PC端**：新增训练数据中心页面；训练数据查询接口；管辖范围权限过滤逻辑
- **小程序端**：训练提交后增加同步后端API调用；网络异常时本地缓存并重试

**优先级建议**：P0（必须）
**预计工时**：10人天（后端数据模型设计2天 + 同步API开发2天 + PC端训练数据中心3天 + 小程序同步逻辑改造2天 + 联调1天）

### 【#ST-02】题库-楼盘关联机制未定义

**优先级**：🔴高
**所属模块**：话术训练
**问题描述**：PRD中提到话术训练的题目与楼盘挂钩，但题库（Questions）与楼盘（Buildings）之间的关联关系完全没有定义。导致：同一题库可以被所有楼盘共用（失去针对性）？还是每楼盘有独立题库（维护成本极高）？经纪人练习时如何匹配到正确题库？

**解决方案**：

1. **定义「楼盘标签」维度的题库共享机制**：
   - 新建 `question_bank_tags` 关联表（多对多）：一个题库可打多个标签（如「刚需盘」「豪宅盘」「商业盘」）
   - 楼盘表新增 `tag_ids[]` 字段（标签数组）
   - 训练任务下发时，系统根据楼盘标签自动匹配关联题库；经纪人也可手动选择题库

2. **建立「楼盘-题库」映射配置页面**（PC端后台）：
   - 支持按楼盘设置专属题库（楼盘维度题库优先于标签匹配）
   - 支持按标签批量设置题库归属
   - 未配置楼盘默认使用「通用题库」（系统预置）

3. **题库优先级规则**：
   - 规则：楼盘专属题库 > 标签匹配题库 > 通用题库
   - 同一标签可关联多个题库（系统随机选一个或按创建时间取最新）

4. **小程序端适配**：
   - 经纪人发起训练时，选择楼盘后自动展示推荐题库
   - 也支持「不限楼盘」使用通用题库

**涉及的数据字段变更**：
- 新增字段：`question_banks` 表新增 `tag_ids[]`（标签数组，非必填）
- 新增字段：`buildings` 表新增 `exclusive_question_bank_id`（专属题库ID，非必填，关联question_banks）
- 新增字段：`question_bank_tags` 表（tag_id, tag_name, description, is_active, created_at）
- 新增字段：`question_bank_tag_relations` 表（question_bank_id, tag_id）
- 修改字段：`training_task` 表新增 `question_bank_id`（任务关联题库ID，非空）

**涉及的流程变更**：
- 新增流程：楼盘题库配置流程（PC端后台）
- 新增流程：训练发起时题库自动匹配流程（小程序端 + 后端）
- 修改流程：训练任务下发流程（任务创建时绑定题库ID）

**两端影响**：
- **PC端**：后台新增楼盘题库配置页（楼盘维度 + 标签维度）；训练任务下发时选择题库
- **小程序端**：训练发起时楼盘选择后展示推荐题库；训练记录关联楼盘ID

**优先级建议**：P0（必须）
**预计工时**：6人天（数据模型设计1天 + PC端题库配置页2天 + 小程序题库匹配逻辑1天 + 后端匹配API 1天 + 联调1天）

### 【#ST-03】训练任务下发机制未定义

**优先级**：🔴高
**所属模块**：话术训练
**问题描述**：PRD提到「下发训练任务」，但下发方式是PC端手动选择经纪人下发，还是系统自动周期性下发，还是按某种规则（如新盘上市自动触发）触发？完全未定义。无法指导开发和测试。

**解决方案**：

1. **定义两种下发模式并存**：

   **模式A：手动下发（PC端）**
   - PC端训练任务管理页 → 新建任务 → 选择楼盘 → 选择题库 → 选择经纪人（或按团队勾选）→ 设置截止时间 → 确认下发
   - 支持批量导入经纪人名单（Excel）
   - 下发后经纪人收到服务通知/模板消息

   **模式B：自动下发（系统规则）**
   - 触发规则配置表（`training_auto_rules`）：支持配置多种自动触发规则
     - 规则1「新盘上市自动下发」：新盘状态变为「在售」时，自动向指定团队/全员下发该楼盘的话术训练任务
     - 规则2「周期性训练」：按月/周向指定团队下发题库训练任务（如每周一9:00）
     - 规则3「考核周期触发」：每月1号自动向本月到期经纪人下发复习任务
   - 规则由系统管理员在PC端配置

2. **自动下发执行机制**：
   - 后台定时任务（JOB）扫描触发规则，满足条件则自动生成训练任务并推送
   - 自动下发的任务标记 `is_auto下发=true`，与手动下发任务区分展示

3. **训练任务状态机**：
   - 草稿 → 已下发 → 进行中 → 已完成/已超时

**涉及的数据字段变更**：
- 新增字段：`training_task` 表新增 `task_type`（manual/auto）、`auto_rule_id`（来源规则ID，非auto时为空）
- 新增字段：`training_auto_rules` 表（rule_id, rule_name, rule_type, trigger_condition, target_scope, question_bank_id, is_active, created_by, created_at）
- 修改字段：`training_task` 表新增 `expire_time`（截止时间）
- 修改字段：`training_task` 表新增 `status`（draft/published/in_progress/completed/expired）

**涉及的流程变更**：
- 新增流程：手动下发训练任务流程（PC端）
- 新增流程：自动下发规则配置流程（PC端）
- 新增流程：定时任务扫描 → 规则匹配 → 自动下发 → 通知推送

**两端影响**：

- 新增流程：跨端数据一致性校验流程（T+1批次校验）

### 【#MM-02】小程序端物料筛选维度未说明

**优先级**：🟡中
**所属模块**：物料管理
**问题描述**：小程序端物料中心虽然有筛选功能，但PRD未定义具体支持哪些筛选维度。物料量大（几百张海报+视频）时，筛选维度不清晰会导致经纪人找不到需要的物料。

**解决方案**：

1. **定义筛选维度**：
   - **按物料类型**：海报 / 视频 / 楼书 / 户型图 / VR全景 / 其他（单选）
   - **按楼盘**：从楼盘列表选择（与#CM-02共用楼盘数据源）
   - **按发布时间**：不限 / 近7天 / 近30天 / 近3个月 / 自定义时间段

2. **默认排序**：
   - 默认按发布时间倒序（最新发布在前）
   - 支持切换为「按楼盘分组」视图

3. **搜索功能**：
   - 物料名称关键词搜索（模糊匹配）

**涉及的数据字段变更**：
- 无额外字段变更（物料表已有 material_type 和 created_at 字段）

**涉及的流程变更**：
- 修改流程：物料列表查询流程（增加多维度筛选参数）

**两端影响**：
- **PC端**：物料列表已有筛选功能，需对标小程序筛选维度确保一致
- **小程序端**：物料中心列表页增加类型+楼盘+时间三维筛选

**优先级建议**：P2（优化）
**预计工时**：1.5人天（小程序筛选UI改造1.5天）

### 【#MM-03】物料海报小程序码指向未定义

**优先级**：🟡中
**所属模块**：物料管理
**问题描述**：物料海报（如项目宣传海报）上通常会有小程序码，但PRD未定义这个小程序码扫码后应该跳转到哪里。是跳转到楼盘详情页？项目主页？还是其他落地页？不同场景需要不同的跳转逻辑。

**解决方案**：

1. **定义物料小程序码跳转规则**：
   - **按物料来源区分**：
     - 楼盘关联物料（如XX楼盘宣传海报）→ 扫码跳转至该楼盘详情页
     - 品牌宣传物料（无特定楼盘）→ 跳转至品牌馆/楼盘列表首页
   - **跳转参数**：后端生成带参数小程序码（scene参数包含物料ID+楼盘ID+来源标识）

2. **PC端生成带参小程序码**：
   - 物料详情页增加「下载带参小程序码」按钮
   - 后端调用微信「获取小程序码」接口，生成带物料专属scene参数的二维码图片
   - 下载的二维码图片可印刷使用

3. **小程序端扫码落地页**：
   - 小程序识别scene参数 → 查询物料信息 → 渲染落地页
   - 落地页展示物料内容 + 关联楼盘信息 + 「获取更多物料」入口

**涉及的数据字段变更**：
- 修改字段：`materials` 表新增 `miniapp_qr_scene`（小程序码scene参数值，生成时写入）
- 新增字段：`material_qr_log` 表（log_id, material_id, scene_value, scanned_at, user_id）用于物料传播数据统计

**涉及的流程变更**：
- 新增流程：带参小程序码生成流程（PC端）
- 新增流程：小程序扫码落地页解析流程

**两端影响**：
- **PC端**：物料详情页增加「下载带参小程序码」功能；二维码生成管理
- **小程序端**：扫码识别 → 落地页渲染流程

**优先级建议**：P2（优化）
**预计工时**：4人天（后端带参码生成1.5天 + PC端下载按钮1天 + 小程序落地页1.5天）

### 【#NM-02】小程序端新盘列表入口未定义

**优先级**：🟡中
**所属模块**：新盘管理
**问题描述**：PC端新盘管理功能完整，但小程序端经纪人如何找到新盘列表完全没有定义。经纪人不知道在小程序哪里能看到新盘信息，导致新盘信息无法触达一线。

**解决方案**：

1. **小程序端新盘列表入口**：
   - 方案A（推荐）：「工作台」→「新盘速览」Tab（新盘列表 + 筛选）
   - 方案B：首页顶部「热门新盘」Banner，点击进入列表
   - 方案C：在楼盘搜索功能中提供「新盘优先」筛选项

2. **新盘列表内容**：
   - 卡片展示：新盘名称、区域板块、主推户型、参考价格、开盘状态（「即将开盘」「在售」「售罄」）、发布时间
   - 支持按区域、价格、开盘状态筛选
   - 点击卡片跳转至楼盘详情页

3. **新盘上线通知**：
   - 楼盘状态变为「在售」时，已订阅该区域或全区域的经纪人收到服务通知

**涉及的数据字段变更**：
- 新增字段：`buildings` 表新增 `miniapp_display_status`（小程序展示状态：show/hide）
- 修改字段：`buildings` 表已有 `status`（在售/待售/售罄）可复用于小程序筛选

**涉及的流程变更**：
- 新增流程：新盘上线 → 推送通知 → 小程序列表更新

**两端影响**：
- **PC端**：新盘管理页增加「小程序展示」开关（控制小程序端是否展示）
- **小程序端**：新增新盘列表入口页（新盘速览Tab）

**优先级建议**：P1（重要）
**预计工时**：3人天（小程序新盘列表页2天 + 新盘上线通知推送1天 + PC端展示开关0.5天 + 联调0.5天）
